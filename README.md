# RealTimeLearningProject


一个用以研究的显式表达的无关模态的实时运行和实时学习的模型


- 本项目的发展目标，是构建一个能够不依赖人类预训练，只需要自己运行，就能实现一切学习和行为目标的模型。
- 由于作者还要兼顾游戏引擎的开发工作，顾不上那么多，哪怕有AI，概念都需要自己亲自梳理，时间精力总是有限的。
- 总而言之，有感兴趣的那就自己下载吧。
- 对了，本工程自动剥离了输入输出模块，作者认为这需要开发者自己去根据手里的硬件/软件/数据环境，自己去单独部署。
- 更多详情，比如说如何开展关于隐式模型的工作，or想要联系作者的可以关注作者的微信公众号《葱葱的练习日记》哈。

## 这是什么

这是一个**显式认知运行时**项目。  
它不是一个传统意义上的“大模型训练框架”，也不是一个简单的 Agent 工具箱。

这个项目尝试做的是：

- 在一个 **GPU状态场** 上组织持续演化的内部状态
- 用显式模块实现**持续学习闭环**，且全程不依赖任何预训练模块
- 把很多通常被隐藏在黑箱参数里的机制，先拆出来、命名、建模、测试
- 为未来更统一的“隐式动力学涌现版本”建立一个可验证的前置研究分支

当前实现中，核心运行时已经把以下闭环明确写成执行流程：

**Routing -> Residual -> Ghost -> ObservationPolicy -> EFE -> Suspension -> Epiphany -> Structure Update** 

这意味着，本仓库的目标并不是“做一个单次前向的模型”，而是尝试构建一个**持续运行、持续调节、持续学习**的认知系统。 

---

## 为什么要先做“显式表达版”

### 一句话版本

因为我们最终真正想要的，并不是长期停留在这些显式模块上。

我们想要的终极形态，是实现一个**统一的动力学 GPU 场**：  
今天在代码里看到的许多显式机制——例如残差、悬置、观测策略、能量门控、结构分裂、记忆分层——未来理想情况下，应该不再以独立 Python 模块的形式存在，而是作为统一场演化中的**自发涌现结构与动力学偏置**出现。

### 但为什么现在还要显式建模？

因为在隐式版本真正可行之前，我们需要一个中间层：

- 它必须**可解释**
- 它必须**可测试**
- 它必须**可拆解**
- 它必须**可争论**
- 它必须允许不同背景的研究者对“概念是否有效”达成最基本的协作共识

换句话说：

> 显式版不是终点。  
> 显式版是为了给隐式版建立“概念地基”和“验证地基”。

如果没有这个显式前置层，很多未来想放进统一动力学场里的机制，会在今天变成无法验证的直觉、无法讨论的隐喻、无法调试的黑箱。

## 当前系统实现了什么

为了帮助理解，可以把当前仓库看成四层。

---

### 1）物理底板 / 状态底板

系统底层有一个 **Large Static Buffer（LSB）**，它是一个 GPU 上的大型状态张量池，用来承载整个系统的核心状态。代码里明确把它描述为 AI 核心状态的“GPU 内存池”，并以 `2048 x 2048 x feature_dim` 的形式组织。缓冲区使用 FP16，并且要求 CUDA / NVIDIA GPU。

这层的意义不是“存模型权重”，而是承载一个**持续演化的内部状态场**。

---

### 2）显式认知主循环

`ContinualLearningRuntime` 是当前显式表达版的核心编排器。  
它负责把认知闭环写成清晰的执行序列：

- 观测策略选择
- 输入嵌入为 Evidence
- 双通道预加载（routing + ghost）
- EFE 代理评分
- 调度与悬置
- 低频 replay / 结构更新
- 监控指标输出

这部分不是“训练脚本”，而是一个**显式学习闭环运行时**。

---

### 3）能量与资源门控

系统不是默认“永远全功率学习”。  
它显式建模了一个 `EnergyKernel`，其中包括：

- `EnergyState`
- `ActionProfile`
- `EnergyGates`

并通过能量状态来决定：

- 是否允许 burst
- 是否允许高精度观测
- 是否允许结构分裂
- 是否允许 snapshot
- 是否允许 replay
- 最大观测预算
- 更新步长和 dt 的偏置

也就是说，这个系统把“资源约束”当成前置性的必要条件，而不是事后优化。 
---

### 4）记忆、结构与局部认知单元

系统使用 `LNCManager` 管理局部认知单元（LNC）的生命周期。  
当前代码已经明确具备这些概念：

- 启动时的 `Genesis`
- L3 -> L2 -> L1 的加载管道
- 从“死数据”到“活智能”的激活过程
- 结构化快照与回滚
- 分层缓存与持久化

代码中甚至直接把这条路径描述为：  
**“将死数据变为活智能”的过程**。 

这部分说明：系统不是固定网络，而是试图把“功能单元、记忆单元、局部结构单元”做成可加载、可保存、可回滚、可固化的对象。

## 这个项目不是什么

为了避免误解，先说清楚：

### 它不是
- 一个即插即用的商业化大模型框架
- 一个成熟收敛的通用智能系统
- 一个简单的多模块 Agent 编排器
- 一个只靠“仿脑口号”驱动的概念工程
- 一个已经完成的隐式动力学场系统

### 它是
- 一个**显式研究分支**
- 一个**可分析、可测试、可争论**的认知运行时
- 一个为未来“隐式表达版”做概念拆解与实验验证的前置工程

---

## 这套系统的核心理念


因此，当前分支选择显式建模一些高阶机制，例如：

- residual：未被解释的剩余
- ghost：异常与残余共振的索引化表达
- observation policy：观测本身作为策略问题
- suspension：不立即整合，而是允许悬置
- shadow branch：保留备选解释和结构分叉
- energy gating：资源受限下的学习与调度
- structural plasticity：结构不是固定的

这些显式模块不是终点，而是**研究工具**。

在未来理想的隐式版本中，我们希望：

- 更少的硬编码模块边界
- 更统一的底层动力学表达
- 更多由 GPU 场演化自发涌现的功能组织
- 更少由人手工指定、更多由状态场本身生成的局部结构与调节行为

## 推荐的理解路径

这个项目的概念耦合度较高。  
不同背景的读者，建议从不同入口进入。

---

### 如果你是偏工程背景

建议优先关注：

1. `main.py`：系统如何装配、启动、运行
2. `continual_learning_runtime.py`：显式学习闭环如何推进
3. `large_static_buffer.py`：GPU 状态池的组织方式
4. `lnc_manager.py`：局部单元如何被加载、保存、回滚
5. `energy_kernel.py`：资源门控如何影响运行模式

你的核心任务不是一开始理解全部理念，而是先搞清楚：

- 状态在哪里
- 循环如何推进
- CPU / GPU 如何分工
- 哪些模块是快循环，哪些是慢循环
- 哪些接口是研究接口，哪些是工程接口

---

### 如果你是偏机器学习 / 大模型背景

建议先放下“这是不是某种 Transformer 替代品”的预设。  
这个项目更接近：

- 持续学习运行时
- 认知架构实验
- 动力系统中的状态组织
- 资源约束下的主动调节

建议优先理解：

- 为什么不是单次前向，而是持续循环
- 为什么感知和行动都被视为控制问题
- 为什么这里要显式建模 residual / ghost / suspension
- 为什么 energy 不是外围优化，而是主循环的一部分

如果你只从“loss / dataset / checkpoint”视角理解它，会非常不适应。  
但如果你熟悉 world model、active inference、predictive processing、continual learning，这个仓库会更容易进入。

---

### 如果你是偏理论 / 认知科学 / 神经科学背景

建议优先阅读理念部分，再看代码。  
因为这里很多模块名并不是为了“工程好看”，而是为了把一些认知过程显式化：

- 何时整合新证据
- 何时只记录痕迹
- 何时允许分叉而不立刻决断
- 何时因资源限制而放弃高成本认知操作
- 何时允许局部结构生长或回滚

代码是这些假设的载体，不是唯一重点。  
你更适合关心的是：

- 这些显式机制是否真的必要
- 哪些应在未来隐式版中被保留
- 哪些只是过渡性的 scaffolding
- 哪些模块之间存在可以压缩到统一动力学表达中的关系

## 开发者需要先接受的门槛

这是一个有门槛的项目。  
门槛主要不在代码量，而在**概念耦合和系统视角**。

你不一定需要同时掌握所有内容，但至少要准备面对下面几类问题：

### 理念层
- 系统是持续运行的，不是一次性推理
- 输入与输出都可以看成主动控制的一部分
- 结构不是固定的，而是可塑、可分层、可迁移的

### 算法层
- 动力系统
- 预测编码 / 主动推理 / 自由能风格目标
- 资源约束与调度
- 持续学习与结构可塑性

### 工程层
- GPU 大张量状态组织
- 多循环运行时编排
- CPU/GPU 桥接
- 长时运行中的状态一致性与调试

如果你希望快速“上手一个库”，这个项目可能不会给你立刻的满足感。  
如果你希望参与一个仍在形成中的研究框架，它会更适合你。

## 当前状态

这是一个**研究中的系统**。

这意味着：

- 接口还会变化
- 模块边界还会调整
- 某些显式结构未来可能被合并
- 某些当前模块未来可能会被证明不必要
- 某些工程路径未来会被替换为更统一的表达

请不要把当前仓库误读为“最终设计定稿”。

更准确的说法是：

> 这是一个正在生长的显式表达分支。  
> 它的价值，在于帮助我们逐步判断：哪些机制值得留下，哪些机制应该在未来隐式版中被统一吸收。

---



它要做的是：

> 先把真正困难的认知机制显式化、可测试化、可争论化；  
> 然后再逐步把这些机制压缩回一个更统一、更强、更接近“涌现智能”的隐式动力学表达中。

如果你对这种方向感兴趣，欢迎加入。
